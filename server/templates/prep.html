LLD / HLD Quick Reference

SOLID Principles

SRP — Each class has one reason to change. OrderService handles order logic, not payment.

OCP — New behavior via extension (new class), not modifying existing code.

LSP — Subtypes must be substitutable for their base type.

ISP — No client should depend on methods it doesn't use. Focused interfaces.

DIP — High-level modules depend on abstractions, not concretions.

Design Patterns Cheat Sheet

Strategy — Multiple algorithms, swap at runtime. → Payment, pricing, sorting.

Observer — One event notifies multiple listeners. → Notifications, events.

Factory — Complex/varying object creation. → Document creator, vehicle types.

Decorator — Add behavior dynamically. → Toppings, middleware.

State — Behavior changes based on internal state. → Vending machine, order lifecycle.

Command — Encapsulate requests as objects. → Undo/redo, task scheduler.

Chain of Responsibility — Multiple handlers process a request. → Logging, validation.

Builder — Construct complex objects step by step. → Query builder.

LLD Framework (6 Steps)

  1. Clarify Requirements (3-5 min) — Scope, actors, constraints, out-of-scope.
  2. Identify Entities (5-7 min) — Nouns → structs. Relationships: has-a, is-a, uses-a.
  3. Define Interfaces (5-8 min) — Public API, abstraction, decouple components.
  4. Apply Patterns — Only when they solve a real problem.
  5. Walk Through Use Case (5 min) — Trace one flow end-to-end.
  6. Discuss Extensibility (5 min) — How does it handle new requirements?

HLD Framework (5 Steps)

  1. Requirements (5 min) — Functional + Non-functional (scale, latency, consistency).
  2. Capacity Estimation (3-5 min) — DAU → QPS → Storage → Bandwidth.
  3. High-Level Architecture (10-15 min) — Components, data flow, happy path first.
  4. Deep Dive (15-20 min) — DB schema, API design, sharding, caching, consistency.
  5. Failure Handling (5 min) — Circuit breaker, retry, monitoring, deployment.

DB Selection Guide

Transactional → PostgreSQL/MySQL (ACID, JOINs)

High write, flexible schema → MongoDB (horizontal scaling)

Time-series → InfluxDB/TimescaleDB

Cache/session → Redis (sub-ms reads, TTL)

Graph/relationships → Neo4j

Full-text search → Elasticsearch

Blobs/files → S3/GCS

Caching Strategies

Cache-Aside — App checks cache, on miss reads DB. Most common.

Write-Through — Write to cache + DB simultaneously. Strong consistency.

Write-Behind — Write cache only, async to DB. High throughput.

Read-Through — Cache manages DB reads on miss.
